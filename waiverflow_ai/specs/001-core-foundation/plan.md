## Summary

Build the core waiver flow management platform for GCs. The system generates,
sends, and tracks lien waiver requests across subcontractors per pay cycle,
validates returned documents via OCR, tracks COI compliance, auto-escalates
overdue items, and provides a consolidated pay-app readiness dashboard. All
data is tenant-isolated with persistent OIDC accounts for both GC and
subcontractor users.

## Technical Context

**Language/Version**: .NET 10 / C# 14 (backend services); Python 3.12 (AI
inference microservice)

**Primary Dependencies**: ASP.NET Core, MediatR, FluentValidation, EF Core,
MassTransit, Semantic Kernel (.NET), LangChain/LlamaIndex (Python), OpenTelemetry

**Storage**: PostgreSQL (operational store + pgvector for document embeddings);
Redis (caching, CQRS read models, rate-limit counters)

**Testing**: xUnit + FluentAssertions (C#); pytest (Python)

**Target Platform**: Linux containers on Kubernetes (AKS), modern web browsers

**Project Type**: Web application — multi-service backend (bounded contexts) +
Python AI service + responsive web frontend

**Performance Goals**: API reads p95 < 200 ms, API writes p95 < 400 ms;
AI first-token < 1.5 s, grounded answer < 6 s

**Constraints**: Multi-tenant row-level security; idempotency keys on commands;
transactional outbox for event publication; all services stateless; external
integrations behind Anti-Corruption Layers; comprehensive structured logging
on every function per file per day

**Scale/Scope**: Multi-tenant micro SaaS; initial target ~100 tenant projects
with 20+ subcontractors each

## Frontend Architecture

**Framework**: Angular 19 with standalone components, no NgModules.

**Pages & routing**:

| Route | Component | Role |
|-------|-----------|------|
| `/login` | LoginComponent | Role selection (dev mode) |
| `/projects` | ProjectsComponent | GC accountant: list/create projects |
| `/projects/:id/subs` | SubsComponent | GC accountant: manage subs, CSV import, pay cycle |
| `/projects/:id/compliance` | ComplianceComponent | GC accountant: COI dashboard |
| `/projects/:id/pay-readiness` | PayReadinessComponent | GC accountant: readiness, export, override |
| `/submit/:waiverId` | SubmitComponent | Sub admin: upload signed waiver |
| `/coi` | CoiComponent | Sub admin: upload COI |

**Services**: `ApiService` wraps all `/api/v1/*` HTTP calls. `AuthService`
manages dev-mode login state in localStorage.

**Styling**: Custom CSS with CSS custom properties (design tokens). No external
UI library — keeps the bundle small and avoids dependency churn.

**Development**: `ng serve --proxy-config proxy.conf.json` proxies `/api/*` to
the .NET backend on port 5000.

## Logging Architecture

Every function in every file logs its entry, key decisions, and exit in
structured JSON format. Logs are organized per file per day.

**Format (all services)**:

```json
{
  "timestamp": "2026-06-26T10:30:00.123Z",
  "level": "info|warn|error|debug",
  "service": "WaiverFlow.DocumentRequests",
  "file": "Services/ProjectService.cs",
  "function": "CreateProject",
  "tenantId": "guid",
  "userId": "guid",
  "correlationId": "guid",
  "message": "Project created",
  "elapsedMs": 45,
  "payload": { }
}
```

**CRUD create payload logging**: Every entity create operation writes a
separate JSON file named `{EntityId}-{yyyy-MM-dd}.json` containing the full
create payload. Files are stored per service under a `logs/payloads/` directory.

Example filename: `logs/payloads/WaiverFlow.DocumentRequests/project_3a1f2c-2026-06-26.json`

```json
{
  "entityType": "Project",
  "entityId": "3a1f2c...",
  "createdAt": "2026-06-26T10:30:00.123Z",
  "tenantId": "guid",
  "createdBy": "guid",
  "payload": { "name": "Riverfront Tower", "description": "Phase 1" }
}
```

**Implementation**:
- .NET: Serilog sink writing JSON to daily rolling files per service
- Python: structlog writing JSON to daily rotating files
- CRUD payload files: dedicated `PayloadFileSink` / `PayloadFileHandler` that
  writes one JSON object per file with `{EntityId}-{yyyy-MM-dd}` naming
- All log files include tenantId, userId, and correlationId for tracing
- Log level is configurable per service via environment variable
- Audit-critical operations (FR-015) use the audit log table; function-level
  logs go to file
